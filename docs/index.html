Local-first canonical game library

One canonical game library for every place your games live.

MGA merges storefronts, ROMs, cloud runtimes, network shares, removable drives, metadata providers, media, files, and achievements into source-backed canonical game pages that stay local by default.

Most launchers start from a storefront or a folder. MGA starts from the game identity. Every detected source, provider match, file location, achievement system, and runtime remains visible, so you can understand, launch, fix, and curate the game instead of trusting a hidden match.

What makes MGA different

Turn fragmented game records into one canonical local library.

Canonical game merge

One canonical game can absorb multiple source records, provider matches, files, and media instead of leaving you with duplicate launcher rows.

Source-backed versions

Multiple concrete versions can live under one canonical game, with per-source launch options and achievement sets kept visible.

Source provenance

MGA keeps every integration visible as source-backed evidence, including source records, provider IDs, resolver matches, and attached files.

Local-first ownership

Your database, media cache, config, and plugins stay on your machine unless you explicitly add sync integrations.

Manual review and repair

Detection failures stay visible and repairable through an explicit review queue with fuzzy, platform-aware provider search instead of being silently hidden behind a bad automatic match.

The problem

Real collections are fragmented by default.

A serious game collection manager has to deal with storefronts, ROM folders, cloud-ready entries, installers, provider metadata, achievements, and files that all describe the same game differently.

Steam Xbox Epic GOG purchases ROMs emulators xCloud browser runtimes SMB / NAS removable drives metadata providers achievement systems local files

Why that matters

  • The same game may exist as storefront data, a ROM root, a cloud launch flag, and an installer package.
  • The best metadata provider may not be the same system that knows launch availability or achievements.
  • When detection is uncertain, you need visible evidence and a repair path, not a hidden importer guess.

Source-backed canonical game pages

The game page is the product centerpiece.

A canonical game page should tell you what the game is, where MGA found it, where it can run, which providers contributed metadata, what files and media are attached, what achievements are known, and what can be fixed manually.

  • what the game is
  • where MGA found it
  • where it can run
  • which providers contributed metadata
  • which source records matched
  • what files and media are attached
  • what achievements are known
  • which version or source each achievement set belongs to
  • what can be manually fixed

Install / download

Windows portable build, local server, local library.

  1. Download the latest Windows portable ZIP from GitHub Releases.
  2. Extract it to a writable folder.
  3. Run Start MGA.cmd.
  4. Open http://127.0.0.1:8900.

MGA runs locally on 127.0.0.1:8900 by default and stores runtime state beside the portable folder unless sync integrations are configured. LAN binding is opt-in through the server LISTEN_IP config.

Is MGA a Playnite alternative?

It can be if you care more about canonical identity and provenance than storefront-first presentation.

Does MGA replace Steam, Xbox, or Epic?

No. MGA is a local game library and launcher layer above those systems.

What happens when detection is wrong?

MGA keeps unresolved records in a visible manual review workflow with fuzzy search and repair actions.
