# ADR-0023: Client-authoritative cross-server installation ownership

- **Status:** Accepted; managed-install ownership slice implemented
- **Date:** 2026-07-18
- **Scope:** MGA Client bindings, install roots, managed-install manifests,
  mutation authorization, release/adoption, and cross-server observations
- **Depends on:** ADR-0006, ADR-0007, ADR-0011, ADR-0017, ADR-0020

## Context

ADR-0020 permits one device/OS-user MGA Client to connect to several MGA
Servers. Each server has an independent database and can otherwise request an
installation into the same user-selected directory. Server-local game IDs and
installation rows cannot prove local filesystem ownership. A directory check
also cannot serialize two simultaneous native installers, and separate game
folders do not isolate Windows product registrations or save locations.

Players must be able to recover from a lost or reinstalled server without
silently granting any server on a trusted LAN authority to delete an existing
installation.

## Decision

### Stable local binding identity

Every client binding has a random, stable `binding_id` generated by MGA Client.
It is distinct from the changeable server URL and from server-issued endpoint
and client-instance identifiers. The URL is an endpoint alias, never proof that
a replacement server is the former owner.

Client config schema 3 adds `binding_id`. Loading schema 1 or 2 generates a
binding ID for every existing binding only after the legacy key and document
are validated, then atomically persists schema 3. Unknown schemas and fields
continue to fail closed.

### Client-local ownership catalog

MGA Client owns a versioned, atomically replaced local catalog under its
per-OS-user data directory. Each managed installation has a random
`local_installation_id`, normalized physical path, install kind and product
evidence, owner binding, lifecycle state, and audit timestamps. Server-local
game/source IDs are correlation hints, not ownership credentials.

All agents in the tray process share one catalog and operation coordinator.
Before mutation, the client atomically reserves the normalized destination and,
when known, native product identity. A conflicting command is rejected with a
typed `installation_conflict`; it is never allowed to race after a check.

### Per-binding roots

The profile/device/per-install path remains a player-selected **base**. New
MGA-managed installations use a visible binding namespace:

```text
<base>\MGA\<safe-server-label>-<short-binding-id>\<game>
```

Internal staging and markers remain inside the binding's namespaced root. The
per-OS-user ownership catalog and cross-process lock live in MGA Client's local
data directory. Installed games are not hidden inside `.mga`. Labels are
cosmetic; the binding ID is authoritative.

Storefront-owned and externally installed applications are shared device state
and do not move under a binding root.

### Manifest and mutation authorization

New archive and GOG manifests carry `local_installation_id` and
`owner_binding_id`. Install, update, repair, cleanup, and uninstall require the
active agent binding, catalog owner, local installation ID, manifest evidence,
and existing root/path boundary to agree. A server cannot supply or override
its binding identity.

Legacy manifests are claimed lazily only when exactly one client binding exists
and their server record and existing safety evidence match. With multiple
bindings, ambiguous legacy installations become `legacy_unclaimed` and require
local confirmation; the client never guesses.

### Lifecycle

Ownership states are:

```text
owned -> disconnected -> released -> owned (adopted)
```

`disconnected` is derived when the owner binding is absent; it does not change
or weaken ownership. Unbinding never deletes games and defaults to preserving
ownership. The user may release one installation or all installations owned by
that binding without deleting files.

A released installation may be adopted by another binding only after local
MGA Client confirmation. Adoption atomically changes catalog and manifest
ownership. It grants only capabilities supported by verified evidence:

- a complete archive manifest is fully manageable;
- a verified native product/uninstaller may be manageable;
- damaged or incomplete metadata remains launch-only until repaired;
- storefront-owned installations remain shared observations and are not
  exclusively adopted.

Replacing a lost server identity is an explicit local ownership-transfer flow.
Reusing the same host, port, or URL is insufficient proof.

### Cross-server observation

The implemented managed-install slice reports sanitized local observations to
every connected binding as one of: managed here, managed by another MGA server,
released, installing here/elsewhere, interrupted, or legacy unclaimed. It does
not disclose another server's URL, profile, token, key, or install path.
Storefront-managed and externally installed discovery remains shared device
state and must use stable storefront/product IDs, package hashes, ROM hashes,
or explicit linkage—never title similarity alone.

An observing server may show availability but cannot mutate another binding's
installation. Launching it requires a one-time local **Use this installation**
grant. When an owner uninstalls or the OS/storefront removes an installation,
the next immediate inventory report updates every connected server; offline
servers reconcile after reconnect.

### Native products and saves

Separate directories do not make duplicate native products safe. The client
serializes known native product identities as well as paths and blocks a second
managed operation when its evidence matches an existing catalog record. The
initial GOG/Inno implementation uses a conservative installer identity; richer
Add/Remove Programs and storefront product linkage remains required before MGA
offers a general **Use existing installation** flow. Storefront uninstall is
explicitly device-global.

Save domains are coordinated separately under ADR-0017. Only one binding may
hold writable sync ownership for the same resolved local save domain. Releasing
or adopting a game does not silently transfer save-sync authority; that is an
explicit, separately confirmed action.

## Security and failure behavior

- An HTTP trusted-LAN server can request release/adoption but cannot complete
  an ownership transfer without local confirmation.
- Catalog and manifest updates use temporary files plus atomic replacement.
- A failure between catalog and manifest writes leaves the installation
  non-mutable until reconciliation repairs the mismatch; it never broadens
  authority.
- Unbinding or losing a server preserves files and ownership history.
- Removing a running installation is rejected until process/operation leases
  are clear.
- Per-user catalogs do not claim another OS user's files. Machine-wide Windows
  registrations are shared observations.

## Migration and compatibility

This decision changes persisted client JSON and managed-install manifests.
Client config migration schema 2 -> 3 is additive and atomic. The ownership
catalog begins at schema 1. Archive manifest schema 3 and executable manifest
schema 4 add owner fields; old schemas retain their existing validation and are
never rewritten without a safe claim/adoption action.

Server migration 26 additively persists `managed_installations_json` on device
inventory snapshots. Existing installation rows remain server-local historical
records and existing inventory JSON remains readable. Migration 26 is already
applied to the local development database and must never be edited; later
changes require migration 27.

Rollback to an older client remains able to see files but must not be used to
mutate schema-3/schema-4 owned installations because older clients do not
understand the ownership fields. Packaging must therefore raise the minimum
client version before server UI enables ownership transfers.

## Acceptance criteria

- Existing config schemas migrate atomically to stable unique binding IDs.
- Two bound servers choosing the same base receive different visible roots.
- Concurrent commands for one path/product cannot both execute.
- A binding cannot uninstall another binding's managed installation even when
  it knows the exact path and server-local IDs happen to match.
- Unbinding preserves catalog ownership and files.
- Release preserves files and makes the record adoptable.
- Adoption requires local confirmation and changes catalog plus manifest
  ownership atomically or fails closed.
- Known native-product conflicts and ambiguous legacy installs fail with
  actionable reasons; richer OS/storefront identity matching remains separate
  follow-up work.
- Tests cover migration, ownership mismatch, reservation collision,
  disconnect/release/adopt, partial-write recovery, and legacy compatibility.
