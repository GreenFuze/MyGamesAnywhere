The MGA model

Canonical game identity first, launcher actions second.

MGA is built for fragmented real-world collections. It does not treat storefronts, ROM folders, metadata providers, achievements, cloud flags, and local files as separate products. It treats them as evidence that should converge on one canonical game.

Source records

Each source integration contributes concrete records, file paths, runtime facts, and raw titles that remain inspectable.

Resolver matches

MGA records which provider matches contributed to the canonical game instead of hiding that decision behind importer magic.

Canonical game page

The game page pulls together metadata, launch availability, media, favorites, achievements, files, source records, and external links.

Concrete versions

Source games remain concrete versions under the canonical page, so play options and achievement sets can show the source/platform they belong to.

Repair workflow

When the resolver cannot prove a match, the record stays visible in manual review with re-detect and fuzzy, platform-aware provider-search actions.

Safer file cleanup

File-backed deletes use source-plugin previews first and require checkbox confirmation before the real delete call is enabled.

Configurable listen IP

The server can bind to loopback, a LAN address, or all IPv4 interfaces while portable releases stay local-only by default.

Profile-owned libraries

Local profiles separate integrations, storefront credentials, scans, favorites, achievements, and library state while plugin binaries and media assets stay global.

Installer and update flow

Windows packages include portable ZIP and installer artifacts. For-me-only installs run as a sign-in process; all-users installs run as an administrator-approved service with LAN defaults, tray startup, firewall setup, rotating file logs, and SHA256-verified self-update flows.

Media cache controls

Settings exposes asynchronous media download status, failed retry controls, and a clear-cache action that removes only local artwork/video files.

What a source-backed canonical game page should answer

  • What is this game?
  • Where did MGA find it?
  • Which providers matched it?
  • What files belong to it?
  • Where can it run?
  • What achievements are known?
  • Which source/version owns each launch option and achievement set?
  • Which links point back to external providers?
  • What needs manual repair if the match is wrong?

Local-first runtime

MGA behaves more like a self-hosted game library on your machine than a cloud-first account dashboard. The local server, local database, local media cache, config, and plugins are part of the architecture, not incidental implementation details. Remote and network-backed browser play can be materialized into a local cache before launch, and EmulatorJS save states can sync through MGA's local save-sync cache.
