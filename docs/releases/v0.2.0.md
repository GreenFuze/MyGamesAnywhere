# v0.2.0 Release Notes

`v0.2.0` is MGA's web-first device-control release after `v0.1.2`. It retires
the experimental desktop application and introduces the standalone, per-user
MGA Client for machine-local work that the browser cannot perform safely.

## Highlights

- **MGA is now web-first.** The React interface served by MGA Server is the
  product UI. The unfinished Avalonia desktop client has been archived in Git
  history and removed from the active source tree and release payload.
- **New standalone MGA Client.** The Go-based client runs once per Windows user,
  requires no permanent administrator rights, stores its Ed25519 endpoint key
  with current-user DPAPI, connects outbound to MGA Server, and installs under
  its own top-level `client/` boundary.
- **Device control plane.** Settings → Devices shows each physical-device / OS-
  user client separately, with authenticated live presence, ready/busy/offline/
  update-required/error states, explicit per-profile permissions, pairing,
  rename/revoke/share controls, command history, ping, and metadata refresh.
- **Top-bar client control.** The web interface always exposes MGA Client state.
  A short-lived signed `mga://start` challenge identifies and wakes the correct
  per-user endpoint; live server presence drives the color. Manage/Owner
  profiles can confirm a typed client-stop command.
- **Optional profile passwords and PINs.** Credentials are verified at profile
  sign-in, include Show/Hide controls and OS-bound recovery commands, and remain
  optional for profiles that do not need protected device authority.
- **Clearer profile and device UX.** Credential management belongs to Profiles;
  device cards are collapsed by default; profile permissions now explain that
  they authorize MGA web profiles, not Windows accounts.
- **Web library polish.** Game cards preserve media composition, landscape art
  receives wider cells, source integrations have distinct badges, and media
  override actions use the corrected API fields.

## Security and behavior

- An endpoint identity is the combination of physical host context, OS user,
  and MGA Client installation. A second Windows user creates a separate client
  endpoint.
- `mga://` is a pairing/wake entry point only. It never carries a reusable
  credential or arbitrary command; commands continue over the authenticated
  server/client channel.
- Device access is explicit: View, Play, Manage, and Owner grants are checked by
  the server for every typed operation. Client stop requires Manage.
- The per-user installer registers `mga://` and starts the client at Windows
  login. Startup is a convenience; authenticated live presence remains the
  authoritative state.

## Upgrade Notes

- Database migration 11 adds `profile_credentials` and `auth_sessions`.
- Database migration 12 adds `device_endpoints`, `device_grants`,
  `device_pairing_challenges`, and `device_commands` plus their indexes.
- Both migrations are additive. Existing profiles, integrations, library data,
  settings, media, and sync rows are preserved.
- The first administrator profile receives the temporary password `changeme`
  only when it has no credential. It must be changed at first protected sign-in.
  Existing credentials are never replaced.
- MGA Client starts at JSON config schema 1; there is no legacy MGA Client state
  to migrate.
- `NO_MIGRATION_NEEDED` for the desktop removal, top-bar association,
  collapsible cards, permission help, credential visibility/recovery, and the
  Profiles/Devices UI separation. These remove unused source or change
  transient/additive UI state without invalidating existing persisted data.
